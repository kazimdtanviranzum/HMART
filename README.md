# HMART — Risk-Sensitive Hierarchical Attention RL for Hospital Transport Robots

This repository contains the code and result data for the paper:

> **Risk-Sensitive Hierarchical Attention Reinforcement Learning for Multi-Objective
> Scheduling and Routing of Autonomous Transport Robots in Hospitals**

HMART (Hierarchical Multi-Agent Reinforcement Learning for Transport) schedules a fleet
of autonomous mobile robots that carry specimens and medicines between hospital wards and
laboratories. It jointly learns request **assignment** and **route improvement**, bounds the
tail of clinical deadline misses with a **Conditional Value-at-Risk (CVaR)** objective, and
coordinates the capacitated elevators with **model predictive control (MPC)**. The problem is
modeled as a multi-objective Markov game and evaluated in a discrete-event simulator of a
four-floor hospital with three elevators.

## What is included

- **`src/neural.py`** — the learned assignment decoder described in Section 4. It embeds each
  candidate request with the global queue context, scores candidates with a shared-weight
  attention scorer, and selects a request with a softmax. Training uses REINFORCE with
  reward-to-go and a batch-normalized baseline; routing, batching, and MPC reuse the same
  modules as the heuristic HMART policy.
- **`data/results.json`** — the aggregated results that back every table and figure in the
  experiments section (mean and standard deviation over 20 seeds for the dynamic experiments).
- **`data/neural_eval.json`** — the metrics of the trained policy reported in Table 2 and Table 9.

## Before the repository is made public — files to add

The paper (Appendix C) describes a small set of scripts that together reproduce every result.
`src/neural.py` imports the simulator with `from sim import ...`, so the simulator module is
required for the training code to run. Please add the following files to `src/` before publishing:

- `sim.py` — the discrete-event hospital simulator (builds the hospital graph, generates
  requests, runs each policy, and records the metrics; exposes `Hospital`,
  `generate_requests`, `simulate`, `cvar`, `CLASSES`).
- `exact.py` — the exact mixed-integer program for the static case (Section 6.1).
- `experiments.py` — the driver that runs all methods and writes `data/results.json`.
- `figures.py` — the script that produces the figures from the result data.

Once these are added, the repository will contain the full pipeline referenced in the paper.

## Reproducing the results

The trained policy is produced and evaluated with:

```bash
cd src
python neural.py
```

This trains the assignment policy with 8 robots at a stable load over a four-hour horizon
(120 updates, 8 episodes per update), then evaluates the frozen policy on the unseen setting
of 12 robots at heavy load over an eight-hour shift across 20 seeds. The evaluation metrics
are written to `neural_eval.json` and match the HMART-RL row of Table 2 and Table 9
(total weighted tardiness of about 40, STAT service level of about 0.965, CVaR-95 of about 0.93).

The full experiment suite (`data/results.json`) is regenerated by running the experiment driver
once it is added (see the section above).

## Key settings

| Setting | Value |
| --- | --- |
| Floors / service points / elevators | 4 / 32 / 3 (one express) |
| Class mix (STAT / urgent / routine) | 10% / 30% / 60% |
| Deadline allowance (minutes) | 15 / 45 / 120 |
| Class weights | 8 / 3 / 1 |
| Robot capacity | 3 |
| Shift length | 8 hours |
| Fleet sizes | 4, 8, 12, 16, 24 |
| Seeds per dynamic result | 20 |
| CVaR tail level | 0.95 |

The full hyperparameter list is in Appendix A of the paper.
